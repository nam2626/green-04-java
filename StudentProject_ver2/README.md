# 학적 관리 프로그램 Version 2 (Refactoring Project)

이 프로젝트는 `StudentProject_ver1`의 단순한 구조를 객체 지향의 핵심 원칙을 적용하여 더 체계적으로 업그레이드한 버전입니다.

---

## 🚀 왜 구조를 바꿨을까요? (ver1 vs ver2)

### **기존 구조 (ver1)의 문제점**
- **Main 클래스의 과부하**: `StudentMain`이 메뉴 출력, 입력 받기, 어떤 기능을 실행할지 결정하는 일까지 너무 많은 일을 혼자 처리했습니다.
- **유지보수의 어려움**: 기능이 100개로 늘어난다면 `Main` 클래스의 `switch-case`문은 수천 줄이 되어 코드를 읽기조차 힘들어집니다.
- **분업의 부재**: 요리사 한 명이 주문도 받고, 요리도 하고, 서빙도 하고, 계산도 하는 상황과 같습니다.

### **새로운 구조 (ver2)의 도입**
- **역할 분담**: 각 클래스가 "딱 한 가지 종류의 일"만 잘하도록 쪼개어 효율성을 높였습니다.

---

## 📂 핵심 구성 요소와 역할

### 1. **Controller (컨트롤러) : "분야별 전문가"**
- **역할**: 사용자의 요청(메뉴 선택) 하나하나를 전담해서 처리합니다.
- **왜 필요한가?**: 등록 전문가(`StudentInsertController`), 조회 전문가(`StudentAllPrintController`) 등으로 나누어 놓으면, 나중에 등록 기능에 버그가 생겼을 때 '등록 전문가' 코드만 열어보면 됩니다. 다른 코드를 건드릴 위험이 사라집니다.

### 2. **HandlerMapping (핸들러 매핑) : "지배인/안내원"**
- **역할**: 사용자가 입력한 번호를 보고 "그 일은 이 전문가(Controller)가 담당이야!"라고 연결해 줍니다.
- **왜 필요한가?**: `Main` 클래스가 일일이 전문가를 찾을 필요 없이, 안내원(`HandlerMapping`)에게 물어보기만 하면 전문가를 데려다주기 때문에 `Main` 코드가 매우 깔끔해집니다.

### 3. **StudentService (싱글톤 서비스) : "중앙 데이터 창고"**
- **역할**: 모든 학생 데이터가 저장된 배열을 관리합니다.
- **특징**: `Singleton` 패턴을 사용하여 프로그램 전체에서 **딱 하나의 창고**만 존재하도록 보장합니다. 등록 전문가든 조회 전문가든 모두 같은 창고를 이용해야 데이터가 꼬이지 않기 때문입니다.

---

## 💡 비유로 이해하기 (음식점)

- **StudentMain**: 식당의 전체적인 영업 시간(무한 루프)을 관리합니다.
- **HandlerMapping**: 손님이 "스테이크 주세요"라고 하면 "양식 담당 요리사"를 호출해 주는 **주방 지배인**입니다.
- **Controller**: 지배인의 호출을 받아 실제 요리를 완성하는 **각 파트 요리사**입니다.
- **StudentService**: 요리사들이 공통으로 재료를 꺼내 쓰고 저장하는 **대형 냉장고**입니다.

---

## 📈 발전 방향
현재 등록(1번)과 전체 조회(5번) 기능이 컨트롤러 패턴으로 전환되었습니다. 나머지 기능(삭제, 수정 등)도 순차적으로 전담 전문가(Controller)를 만들어 독립시킬 예정입니다.
